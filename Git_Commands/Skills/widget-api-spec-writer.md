# Skill: widget-api-spec-writer

## Purpose

Writes or updates a widget's backend API spec once its design is locked — the doc that gets handed to developers and pasted into Confluence. Follows `Step 5 - API documents/How to Write a Widget API Spec.md`'s structure exactly, and checks the real existing API first (via the classic/modern comparison docs) rather than guessing at tables or fields.

## Hard rules it follows

- **Gated on Step 4.** Won't start a new spec for a widget that isn't ✅ Done in `Step 4 - Widget Final Design/00 - INDEX.md` — says so plainly and stops if asked to jump ahead.
- **Always produces a matched pair** — `<Widget> - API Spec.md` (for future AI sessions) and `<Widget> - API Spec (Confluence).html` (styled to match the existing examples, generated by script, never hand-written).
- **Updates route through the same accept/reject diff-review widget** as `widget-final-check-audit` — once a spec has gone out to developers, a silent rewrite isn't acceptable.
- **Never marks a spec "done" on its own.** Only removes the `Status: DRAFT — not final` line (never replaces it with "APPROVED") when the project owner explicitly says the spec is done.
- **Keeps `Step 5 - API documents/00 - INDEX.md` in sync automatically** as a side effect of creating, updating, or approving a spec.

## How to run it

Ask in plain language, e.g.:
- "Write the API spec for Pension Plans."
- "Update the Deposit Accounts API spec — the DB table changed."
- "Mark the Payroll Distributions spec as done."

## Files

- `SKILL.md` — the full instructions.
- `scripts/build_confluence_html.py` — converts the `.md` spec into the Confluence-styled `.html`.
- `scripts/build_diff_review.py` — same accept/reject widget mechanism as the audit skill.
