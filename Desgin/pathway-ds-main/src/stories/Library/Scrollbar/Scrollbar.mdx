One overlay scrollbar for the whole design system. Wrap any internal scroll region in `<Scrollable>` and it hides the native OS scrollbar and draws a single slim, semitransparent thumb — so scrolling looks and behaves **identically on macOS, Windows, iOS, and Android**, hugs the right edge, and never takes layout width or shifts padding.

Use it for **every** internal scroll surface in Pathway — the SideNav menu, dropdown and popover panels, dialogs, long lists. Don't fall back to a raw `overflow-y: auto` with native scrollbars: that bar is chunky and space-taking on Windows, thin-overlay on macOS, and unstylable on mobile — three different looks, and on Windows it shifts your content.

## Anatomy

```
div.pds-scrollable                      position: relative   (the positioning context)
├── div.pds-scrollable__view            overflow-y: auto; native bar hidden
│   └── {children}                      your tall content (put content padding HERE, via viewStyle)
└── div[aria-hidden="true"]             the custom thumb — position: absolute; right: 2px
                                        width 6px · radius 6px · semitransparent · overlay only
```

The native scrollbar is hidden once, globally, by a single injected rule (`<style id="pds-scrollable-base">`) scoped to `.pds-scrollable__view` — scrollbars can't be hidden via inline React styles, so this is the one stylesheet the component relies on. The underlying content still scrolls natively (wheel, keyboard, touch, drag); only the *visible* bar is ours.

---

## The thumb always hugs the right edge

The thumb is positioned `2px` from the **right edge of the `<Scrollable>` wrapper** — in every state, at every breakpoint, on any consumer. For it to land on the *visible* edge of your scroll surface, the wrapper must reach that edge.

> **IMPLEMENTATION RULE: content padding goes on `viewStyle`, never on the wrapper.**
> Put right-edge padding *inside* the scroll view (`viewStyle`), not on an outer padded container. If a consumer wraps `<Scrollable>` in a container with right padding, the thumb floats inward by that amount. The SideNav rail showed this: the nav's 12px `border-box` padding pushed the thumb ~14px in on the 72px rail. Fix = bleed the scroll region to the true edge (`marginRight: -pad`) and re-add the inset inside `viewStyle` (`paddingRight: pad`). The thumb then hugs the edge in both the expanded and collapsed states.

## Behaviour

- **Overlay, never layout-affecting.** Absolutely positioned over the content's right edge. Zero layout width — never pushes content, changes padding, or resizes the container.
- **Reveal on activity.** Hidden (opacity 0) when idle; fades in (240 ms) on hover or while scrolling; fades out ~900 ms after scrolling stops. On touch it appears while scrolling.
- **Draggable**, and the underlying content also responds to wheel / keyboard / touch as normal.
- **Auto-sizing.** Thumb length = viewport ÷ content ratio (min 28 px); recomputes on content and container resize via `ResizeObserver`.

## Tokens & constants

Every colour and unit resolves through a **semantic** Pathway token (exported as `SCROLL` from `scrollbar.jsx`) — no primitives, no hardcoded hex/px. Colour uses the `fill.static.neutral` overlay semantics; geometry uses the semantic `layout-units` tokens.

| Constant | Semantic token | Resolved | Meaning |
|---|---|---|---|
| `thumbWidth` | `--semantic-layout-units-padding-xtight` | 6 px | thumb thickness |
| `thumbRadius` | `--semantic-layout-units-cornerradius-full` | pill | fully rounded |
| `thumbMin` | — | 28 px | min length (numeric in JS) |
| `gutter` | `--semantic-layout-units-padding-xxxtight` | 2 px | inset from the right edge |
| `thumbRest` | `--semantic-color-light-mode-scrim-faint` | `rgba(17,17,17,0.16)` | resting overlay (black 16%) |
| `thumbHover` | `--semantic-color-light-mode-scrim-light` | `rgba(17,17,17,0.30)` | hover / drag overlay (black 30%) |
| `thumbEdge` | `--semantic-color-light-mode-fill-static-neutral-light` @ 35% | white, 35% | hairline glass edge (`color-mix`) |
| `thumbBlur` | — (effect) | `blur(8px) saturate(180%)` | backdrop-filter (liquid glass) |
| `fadeMs` | — | 240 ms | reveal / hide transition |
| `idleHideMs` | — | 900 ms | hide delay after scrolling stops |

> **Note:** the thumb uses the **mode-aware** `scrim` overlay family — `scrim/faint` is black 16% in light mode (a faint dark sliver) and white 16% in dark mode, so it stays visible against either surface with no component change. Hover steps up to `scrim/light` (30%). `scrim/faint` was added for faint overlay controls, below the existing light/subtle/base scrim steps.

## Accessibility

The underlying content scrolls natively, so keyboard scrolling, focus order, and screen-reader behaviour are unchanged. The thumb is decorative (`aria-hidden="true"`) — a visual affordance, not the scroll mechanism. Reduced-motion users still get the bar; only the fade is cosmetic.

---

## Constraints

1. **Never use a raw native scrollbar for an internal scroll region.** Use `<Scrollable>`. Native bars differ per OS and shift content on Windows.
2. **Content padding belongs on `viewStyle`, never on the wrapper** — see the implementation rule above, or the thumb floats off the edge.
3. **The wrapper must be allowed to size to its container** (`minHeight: 0` inside a flex column) so the inner view can overflow and scroll.
4. **The thumb is decorative.** Never make it the only scroll mechanism; native scrolling always works underneath.
5. **Constants are the contract.** `thumbWidth`, `gutter`, colours, and timings are shared across the system — don't override them per-component.

## Why no Figma node

Every other Pathway component has a Figma source of truth. The scrollbar deliberately doesn't. It exists *because* native scrollbars can't be made consistent across platforms — its entire job is runtime behaviour (hide the OS bar, draw an overlay thumb, fade on activity, hug the edge). None of that is a static visual a Figma frame can hold, and there are no variants or design properties to bind. So this component's source of truth is the spec + the `.jsx`, and the token-reconciliation flow that keys off a Figma node simply doesn't apply here.
