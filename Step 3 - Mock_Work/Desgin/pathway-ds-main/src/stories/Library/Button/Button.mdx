The Button is Pathway's primary action trigger. It is the component a user interacts with to commit a change, advance a flow, or execute a command. Every interactive surface that causes something to happen should be a Button — not a styled div, not an anchor with an `onclick`, not a custom element.

Three visual styles express the hierarchy of weight on a page: **Fill** (the strongest signal — use once per primary action), **Outlined** (secondary weight — supporting actions adjacent to a primary), and **Naked** (lowest visual weight — inline links and tertiary actions that shouldn't compete for attention). Four semantic types handle intent: **Primary** (brand blue — the default), **Secondary** (neutral muted), **Tertiary** (lightest), and **Negative** (destructive or error actions — delete, revoke, sign out).

The Button does not carry navigation semantics. It fires an `onClick`. If pressing it navigates the user to a different route, use an anchor or your router's `<Link>` component styled with Button tokens instead — and accept the accessibility implications.

## Anatomy

```
<button type="button">                    ← outer — touch-target zone (min 48×48px)
  <span class="pw-button__container">    ← Container.Main — visible surface (bg, border, radius)
    <span class="material-symbols-rounded  pw-button__icon--leading">  ← optional leading icon
    <span class="pw-button__label">      ← label text
    <span class="material-symbols-rounded  pw-button__icon--trailing"> ← optional trailing icon
  </span>
</button>
```

The outer `<button>` carries all interactive semantics (`aria-label`, `aria-busy`, `disabled`, `onClick`) and acts as the WCAG 2.5.5 touch target (min 48 × 48 px). It is transparent — no fill, no border. The visual surface lives entirely in `Container.Main`: background, border, border-radius, focus ring via `box-shadow`, gap, and padding.

This two-layer approach means the visible button surface can be exactly the size dictated by its content while the interactive zone always meets the 48 px minimum. The 6 px gap between the outer and inner elements is the mandatory touch clearance.

## States

<Canvas of={Stories.StateMatrix} />

All 3 styles × 4 types × 6 interaction states rendered as live components. The state machine is straightforward: `disabled` and `loading` are terminal (they override all interaction); otherwise, `pressed` > `hovered` > focused > `base` in priority order.

**Loading** deserves special mention: it is not just a visual state — it is a semantic one. When `loading={true}`, all content (label, icons) is replaced by a spinner. `aria-busy="true"` is set on the button. The button is implicitly disabled to prevent double-submission. The spinner inherits its colour from the button's current icon token via `currentColor`, so it always matches the expected icon colour for that style/type combination.

## Sizes

<Canvas of={Stories.AllSizes} />

Four sizes: **L** (18 px label), **M** (16 px, default), **S** (14 px), **XS** (12 px). All sizes share the same 8 px gap between icon and label and the same 8 px border radius. Padding scales:

XS uses a 12 px label (`label.button.xs`) and a 16 px icon wrap. It is defined in Figma for **Primary / Secondary / Negative** across Fill / Outlined / Naked — there is **no XS Tertiary** variant. The touch target (48 × 48 px minimum) applies at all sizes, XS included.

## Icon variants

<Canvas of={Stories.IconVariants} />

Icons use Material Symbols Rounded, sized to the button's icon slot (`26 px` for L, `24 px` for M, `20 px` for S). The slot wrapper constrains the icon to that square; the icon renders at `18 / 16 / 14 px` respectively.

For **icon-only buttons** (`showText={false}`), `ariaLabel` is mandatory — without it, the button has no accessible name and fails WCAG 4.1.2.

## Responsive behaviour

The Button has no breakpoint-specific layout changes — it is an inline element sized by its content. The touch target floor (48 × 48 px minimum, enforced via `min-height`/`min-width` on the outer `<button>`) applies at all viewport sizes. Producers of button-containing layouts should ensure adequate spacing between adjacent buttons at narrow widths.
